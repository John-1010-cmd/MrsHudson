## 二、AI_ARCHITECTURE.md 修正详情

### 🔴 严重问题修正

#### 1. L3 完整AI层的调用语义澄清

**问题描述**: 调用链步骤 ③ 对 L3 的描述存在歧义，容易让读者误以为会触发两次完整 AI 调用。

**决策确认**:
- L2 轻量AI层 = 轻量模型（如 Claude Sonnet）
- L3 完整AI层 = 完整模型（如 Claude Opus）
- L3 仅做意图分类（约 50-100 tokens），不生成最终回复
- 步骤 ⑧ 才是生成调用（500-2000 tokens）

**修正内容**:
```
§5.1 调用链步骤 ③：
- 明确 L2 调用轻量模型，L3 调用完整模型
- 补充 token 消耗说明（50-100 tokens）
- 添加注释：与步骤 ⑧ 的生成调用独立

§7.2 意图路由三层架构：
- 新增"三层模型选择说明"表格
- 对比四个阶段的 token 消耗
- 明确 L3 只做意图分类，不生成回复
```

#### 2. QueryNormalizer 归一化策略统一

**问题描述**: §7.3 示例使用占位符方案（`{current_date}`），但 intent-vector-cache-design.md 使用具体日期方案，两者不一致。

**决策确认**: 采用**具体日期方案**
- "今天" → "2026-03-23"
- "明天" → "2026-03-24"
- 优点：查询逻辑简单，向量语义精确
- 缺点：缓存按日期隔离，跨日期不复用

**修正内容**:
```
§7.3 意图向量缓存：
- 示例改为："今天天气怎么样" → "2026-03-23天气怎么样"
- 补充"归一化策略说明"章节
- 说明设计权衡：虽然跨日期复用率降低，但向量语义更准确
```

## 三、SSE_TTS_UNIFIED_SPEC.md 修正详情

### 🔴 严重问题修正

#### 1. §4.2 token_usage 注释歧义

**问题描述**: §4.2 注释"token_usage 事件省略"容易让读者误以为该场景不发 token_usage，但实际上这是"模型不支持 thinking"，不是缓存命中，应照常发送。

**修正内容**:

§4.2 不含思考过程：

**问题描述**: `currentSseClient = null` 放在 `await sseClient.stream()` 之后，快速切换会话时可能被旧流的 finally 清空。

**修正内容**:

§3.6 前端切换会话实现要点：
- 补充身份校验：
  if (currentSseClient === sseClient) {
    currentSseClient = null
  }
- 添加注释："防止竞态：只有当前 client 仍是活跃 client 时才清空"

## 五、关键架构决策确认

### 1. 意图路由三层模型

| 层级 | 模型类型 | Token 消耗 | 用途 |
|------|---------|-----------|------|
| L1 规则层 | 无 AI 调用 | 0 | 关键词匹配 |
| L2 轻量AI层 | 轻量模型（如 Claude Sonnet） | 20-50 tokens | 简单意图识别 |
| L3 完整AI层 | 完整模型（如 Claude Opus） | 50-100 tokens | 复杂意图分类 |
| 步骤 ⑧ 生成调用 | 完整模型 + 完整上下文 | 500-2000 tokens | 最终回复生成 |

### 2. QueryNormalizer 归一化策略

**采用具体日期方案**：
- "今天" → 当前日期（如 "2026-03-23"）
- "明天" → 次日日期（如 "2026-03-24"）
- 权衡：虽然跨日期复用率降低，但向量语义更准确

### 3. VectorStore 过期机制

**Redis 原生 TTL**：
- 通过 EXPIRE 命令设置（如 7 天）
- 自动过期，无需定期 cleanup

### 4. 模型路由策略

**两档路由**：
- 小模型 (moonshot-v1-8k): ≤ 20字 / 问候语
- 标准模型 (moonshot-v1-32k): 其他

### 5. Admin API 路径规范

**统一前缀**：`/api/admin/**`
- 包含缓存、成本、指标、质量模式管理
- 计划迁移 `/api/quality/**` → `/api/admin/quality/**`
