## 一、核心架构概述

### 1.1 什么是 AI Agent？

AI Agent（智能代理）是基于大语言模型（LLM）的智能化系统，具备**推理、规划、工具使用**三大核心能力，能够自主完成复杂任务。

```
传统程序: if-else 规则 → 确定性强，但无法处理复杂场景
LLM Agent: 自然语言推理 → 可处理开放域任务
```

### 1.2 经典架构 (Lilian Weng 架构)

2023年6月，OpenAI 应用研究主管 Lilian Weng 在技术博客中提出 AI Agent 经典架构：

```
┌─────────────────────────────────────────────────────────────────┐
│                     AI Agent 经典架构                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│    ┌─────────────┐                                              │
│    │     LLM     │  ← 决策中枢：推理、逻辑分析、目标拆解          │
│    │ (大模型)    │                                              │
│    └──────┬──────┘                                              │
│           │                                                     │
│  ┌────────┼────────┐                                            │
│  ▼        ▼        ▼                                           │
│ ┌────┐  ┌────┐  ┌────┐                                          │
│ │规划│  │记忆│  │工具│                                          │
│ │模块│  │模块│  │模块│                                          │
│ └────┘  └────┘  └────┘                                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 二、四大核心组件

### 2.1 大模型 (LLM) - 决策中枢

LLM 是 Agent 的"大脑"，承担决策中枢角色：

| 能力 | 说明 |
|------|------|
| **推理 (Reasoning)** | 逐步思考、链式分析 |
| **规划 (Planning)** | 目标拆解、步骤排序 |
| **工具使用 (Tool Use)** | 理解工具描述、正确调用 |

**提示词工程 (System Prompt) 示例：**

```
你是一个专业的AI助手。
你具备以下工具能力：
- get_weather: 查询天气信息
- calendar_event: 管理日历事件
- todo_item: 管理待办事项

你必须：
1. 先理解用户意图
2. 如需调用工具，明确告知用户
3. 工具调用结果必须用于回答用户
```

### 2.2 规划模块 (Planning)

负责将复杂任务分解为可执行的步骤。

#### 2.2.1 任务分解方法

| 方法 | 描述 | 适用场景 |
|------|------|---------|
| **LLM 自身分解** | 让 LLM 直接拆解任务 | 复杂推理任务 |
| **CoT (Chain of Thought)** | 逐步思考，每步记录 | 数学、逻辑问题 |
| **ToT (Tree of Thoughts)** | 多路径探索，剪枝最优 | 创意生成 |
| **ReAct** | 推理 + 行动交替 | 需要外部工具的任务 |

#### 2.2.2 ReAct 模式详解

```
思考 (Thought): 用户问深圳天气
行动 (Action): 调用 get_weather(city="深圳")
观察 (Observation): 返回晴天 25°C
思考 (Thought): 获取到天气信息，可以回答用户
行动 (Action): 给用户返回天气结果
```

**代码实现示例：**

```java
// ReAct 循环伪代码
public String execute(String userInput) {
    while (true) {
        // 1. 推理
        String thought = llm.think(context);

        // 2. 判断是否需要工具
        if (thought.needsTool()) {
            // 3. 执行工具
            ToolResult result = toolRegistry.execute(thought.getToolName(), thought.getArgs());
            // 4. 将结果加入上下文
            context.addObservation(result);
        } else {
            // 5. 直接返回结果
            return thought.getFinalAnswer();
        }
    }
}
```

### 2.3 记忆模块 (Memory)

负责存储和管理任务相关信息。

#### 2.3.1 记忆分层架构

```
┌─────────────────────────────────────────┐
│           长期记忆 (Long-term)           │
│  ┌─────────────┐  ┌─────────────────┐  │
│  │ 用户偏好     │  │ 历史会话摘要     │  │
│  │ (Redis/DB)  │  │ (向量数据库)    │  │
│  └─────────────┘  └─────────────────┘  │
├─────────────────────────────────────────┤
│           短期记忆 (Short-term)          │
│  ┌─────────────────────────────────────┐│
│  │ 对话上下文窗口 (当前会话消息)         ││
│  │ context.window + system prompt      ││
│  └─────────────────────────────────────┘│
└─────────────────────────────────────────┘
```

#### 2.3.2 记忆实现策略

| 策略 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| **完整保留** | 所有消息存入上下文 | 信息不丢失 | 成本高、有长度限制 |
| **摘要压缩** | 定期用 LLM 摘要历史 | 节省 token | 可能丢失细节 |
| **向量检索** | 用 embedding 检索相关历史 | 精准召回 | 需要向量数据库 |
| **混合** | 近期完整 + 久期摘要 + 向量 | 平衡 | 实现复杂 |

### 2.4 工具模块 (Tools)

赋予 Agent 实际的执行能力。

#### 2.4.1 工具调用工作流

```
1. 用户输入 ──────────────────────────────────────┐
                                                    ▼
2. LLM 推理 ──────────────────────────────────────┐
                                                    ▼
3. 解析工具调用 ──────────────────────────────────┐
     │                                             │
     │ 工具名: get_weather                         │
     │ 参数: {"city": "深圳"}                      │
     ▼                                             ▼
4. 执行工具 ──────────────────────────────────────┐
     │                                             │
     │ HTTP 请求 / 数据库查询 / 计算                │
     ▼                                             ▼
5. 返回结果 ──────────────────────────────────────┐
                                                    ▼
6. LLM 生成最终回答 ──────────────────────────────┘
```

#### 2.4.2 工具描述格式 (JSON Schema)

```json
{
  "name": "get_weather",
  "description": "查询指定城市的天气信息",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "城市名称，如北京、深圳"
      }
    },
    "required": ["city"]
  }
}
```

#### 2.4.3 Kimi API 工具调用规范 (重要!)

**必须遵循以下规范，否则会导致 `400 Bad Request` 错误：**

1. **assistant 消息**包含 `tool_calls` 时，`content` 必须为 `null`（不能是空字符串 `""`）
2. **tool 消息**不能包含 `name` 字段（即使为 null）
3. 序列化时需保留 null 值字段（使用 `WriteMapNullValue`）

**正确示例:**
```json
{"role":"assistant","content":null,"tool_calls":[{"id":"call_xxx","type":"function","function":{"name":"get_weather","arguments":"{\"city\":\"深圳\"}"}}]}
{"role":"tool","tool_call_id":"call_xxx","content":"天气结果..."}
```

**错误示例:**
```json
{"role":"assistant","content":"","tool_calls":[...]}  // content 为空字符串
{"role":"tool","tool_call_id":"call_xxx","name":null,"content":"..."}  // 包含 name 字段
```

## 三、六大工程架构模式

### 3.1 路由分发架构 (Router Pattern)

```
                    ┌──────────────┐
                    │  用户输入    │
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐
                    │  路由分类器   │
                    │ (小模型/LLM) │
                    └──────┬───────┘
                           ▼
              ┌────────────┴────────────┐
              ▼                         ▼
       ┌──────────────┐          ┌──────────────┐
       │  小模型处理   │          │  大模型处理   │
       │ (成本低、快) │          │ (成本高、强) │
       └──────┬───────┘          └──────┬───────┘
              └────────────┬────────────┘
                           ▼
                    ┌──────────────┐
                    │  结果返回    │
                    └──────────────┘
```

**典型应用:**
- 简单问答 → 小模型
- 复杂推理 → 大模型
- 代码生成 → 大模型

### 3.2 代理架构 (Agent Pattern)

| 类型 | 说明 |
|------|------|
| **ReAct Agent** | 推理 + 行动交替执行 |
| **Self-Correcting Agent** | 自主纠错机制 |
| **Plan-and-Execute Agent** | 先规划再执行 |

### 3.3 RAG 中心化架构

```
用户查询 ─────────────────────────────────────────┐
                                                 ▼
                    ┌────────────────────────────┐
                    │  问题理解 & 关键词提取       │
                    └──────────────┬───────────────┘
                                   ▼
                    ┌────────────────────────────┐
                    │  向量检索 (知识库)          │
                    │  ──────────────────────     │
                    │  1. embedding(query)        │
                    │  2. similarity search       │
                    │  3. top-k documents         │
                    └──────────────┬───────────────┘
                                   ▼
                    ┌────────────────────────────┐
                    │  上下文增强 (Context        │
                    │  + Retrieved Docs + LLM)   │
                    └──────────────┬───────────────┘
                                   ▼
                    ┌────────────────────────────┐
                    │  LLM 生成答案              │
                    └────────────────────────────┘
```

### 3.4 多智能体协作架构 (Multi-Agent)

```
┌─────────────────────────────────────────────────────────┐
│                    任务输入                             │
└──────────────────────────┬──────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────┐
│                    Agent Supervisor                      │
│              (协调者 - 任务分配与汇总)                    │
└──────────────────────────┬──────────────────────────────┘
                           │
        ┌──────────────────┼──────────────────┐
        ▼                  ▼                  ▼
   ┌─────────┐       ┌─────────┐       ┌─────────┐
   │ Research │       │ Execute  │       │ Review  │
   │  Agent   │       │  Agent   │       │  Agent  │
   └────┬────┘       └────┬────┘       └────┬────┘
        │                  │                  │
        │ 信息收集          │ 执行动作          │ 结果审查
        └──────────────────┴──────────────────┘
                           │
                           ▼
                    ┌──────────────┐
                    │  最终输出    │
                    └──────────────┘
```

### 3.5 MCP (Model Context Protocol) 架构

MCP 是一种标准化协议，用于 AI 与外部系统的交互：

- **工具注册中心**: 统一管理所有可用工具
- **标准化接口**: 统一的调用格式和返回格式
- **可扩展性**: 轻松添加新工具

Mrs. Hudson 采用的就是 MCP 架构模式。

### 3.6 分层架构

| 层级 | 职责 |
|------|------|
| **感知层** | 接收用户输入 |
| **认知层** | LLM 推理 |
| **执行层** | 调用工具 |
| **反馈层** | 结果评估与调整 |

## 四、工程实践关键问题

### 4.1 成本控制

| 策略 | 说明 |
|------|------|
| **流式输出** | 减少等待时间感知 |
| **缓存** | 相同问题用缓存 |
| **小模型优先** | 简单问题不用大模型 |
| **早停** | 答案明确时停止推理 |

### 4.2 错误处理

```java
// 工具执行错误处理
public ToolResult executeSafely(String toolName, String args) {
    try {
        return toolRegistry.execute(toolName, args);
    } catch (Exception e) {
        // 1. 记录日志
        log.error("工具执行失败: {} - {}", toolName, e.getMessage());

        // 2. 返回友好错误
        return ToolResult.error("工具执行失败: " + e.getMessage());

        // 3. 可选: 降级处理
        // return getFallbackResult(toolName);
    }
}
```

### 4.3 安全性

| 风险 | 防护措施 |
|------|---------|
| **Prompt 注入** | 输入清理、系统提示隔离 |
| **工具滥用** | 权限控制、调用频率限制 |
| **数据泄露** | 敏感信息脱敏、审计日志 |
| **越权访问** | 用户身份校验、资源权限检查 |

## 五、优化方向

### 5.1 性能优化

#### 5.1.1 推理加速

| 技术 | 说明 | 效果 |
|------|------|------|
| **流式输出** | token-by-token 实时返回 | 减少等待时间 30-50% |
| **KV Cache** | 缓存注意力机制的 Key/Value | 减少重复计算 |
| **投机解码** | 小模型预测 + 大模型验证 | 加速 2-3 倍 |
| **批量处理** | 合并多个请求 | 提高吞吐量 |

#### 5.1.2 上下文优化

```
优化后:
┌────────────────────────────────────────┐
│ System Prompt (精简至 200 tokens)      │
├────────────────────────────────────────┤
│ 摘要记忆 (Summarized)                  │
│ ─────────────────────                 │
│ 过去 N 轮对话摘要 (300 tokens)         │
├────────────────────────────────────────┤
│ 近期完整记忆 (Recent)                  │
│ 用户消息 N-2 (100 tokens)              │
│ AI 回复 N-2 (150 tokens)               │
│ 用户消息 N-1 (100 tokens)              │
│ AI 回复 N-1 (150 tokens) ← 最新        │
└────────────────────────────────────────┘
```

### 5.2 成本优化

#### 5.2.1 模型路由策略

```
用户输入 ─► 意图分类 ─► ┌─────────────────────────────┐
                      │  判断复杂度 & 任务类型       │
                      └──────────────┬──────────────────┘
                                     │
           ┌─────────────────────────┼───────────────────┐
           │                         │                   │
           ▼                         ▼                   ▼
    ┌─────────────┐          ┌─────────────┐      ┌─────────────┐
    │ 简单查询    │          │ 常规任务    │      │ 复杂推理    │
    │ 小模型     │          │ 中等模型    │      │ 大模型      │
    │ (快、便宜) │          │ (平衡)      │      │ (强、贵)    │
    └─────────────┘          └─────────────┘      └─────────────┘
```

### 5.3 可靠性优化

#### 5.3.1 错误处理与重试

| 错误类型 | 处理策略 |
|----------|---------|
| API 超时 | 指数退避重试 (1s, 2s, 4s, max 3次) |
| 速率限制 | 等待后重试 + 降级 |
| 模型错误 | 返回友好错误 + 记录日志 |
| 工具执行失败 | 返回错误结果 + 建议替代方案 |
| 网络异常 | 降级到缓存/默认响应 |

### 5.4 工具调用优化

#### 并行工具调用

```
串行执行 (慢):                    并行执行 (快):
┌─────────┐    ┌─────────┐       ┌─────────┐
│ 天气 API │ -> │ 日历 API│ ->    │ 天气 API │
└─────────┘    └─────────┘       └─────────┘
   1s            1s               ┌─────────┐
              总计: 3s            │ 日历 API │ (同时执行)
                                   └─────────┘
                                   ┌─────────┐
                                   │ 待办 API │
                                   └─────────┘
                                     总计: 1s
```
