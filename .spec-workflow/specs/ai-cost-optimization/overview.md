# MrsHudson AI调用成本优化方案 - 总体说明

## 一、背景与问题

### 1.1 现状分析

MrsHudson作为AI管家助手，目前每个用户请求都直接调用Kimi API，存在以下问题：

- **成本线性增长**：用户量每增加一倍，AI调用成本同步翻倍
- **重复查询浪费**：用户经常询问相似的天气、日历问题，每次都触发完整AI调用
- **工具调用开销高**：简单的天气查询需要2轮AI调用（判断工具+生成回复）
- **长对话成本爆炸**：对话历史增长导致每次请求的token数线性增加

### 1.2 核心矛盾

| 维度 | 现状 | 理想状态 |
|-----|------|---------|
| 工具查询 | 2轮AI调用 | 零AI调用 |
| 重复问题 | 每次都调AI | 命中缓存直接返回 |
| 长对话 | token线性增长 | 保持恒定 |
| 响应时间 | 2-5秒 | <1秒 |

---

## 二、核心理论

### 2.1 分层拦截理论

AI调用成本优化的本质是**在请求到达AI之前设置多层拦截**，每层处理不同类型的请求：

```
用户请求
    ↓
┌─────────────────────────────────────────┐
│ Layer 1: 语义缓存层                      │
│ 原理：相似问题的向量匹配                  │
│ 拦截率：20-30%                           │
│ 成本：零AI调用                           │
└─────────────────────────────────────────┘
    ↓ 未命中
┌─────────────────────────────────────────┐
│ Layer 2: 意图路由层                      │
│ 原理：规则/轻量AI提取参数                 │
│ 拦截率：85-95%（工具查询）                │
│ 成本：零AI或1轮轻量AI                     │
└─────────────────────────────────────────┘
    ↓ 复杂对话
┌─────────────────────────────────────────┐
│ Layer 3: 对话压缩层                      │
│ 原理：历史消息摘要化                      │
│ 效果：token减少50-60%                    │
│ 成本：降低但不为零                        │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Layer 4: AI调用层（保底）                 │
│ 只有真正需要创造力的对话才到达            │
│ 拦截后比例：5-10%                        │
└─────────────────────────────────────────┘
```

### 2.2 成本递减模型

每层优化都遵循**边际成本递减**规律：

| 优化层 | 实施难度 | 成本节省 | ROI |
|-------|---------|---------|-----|
| 工具缓存 | 低 | 高（重复查询零成本） | ⭐⭐⭐⭐⭐ |
| 意图路由（规则） | 低 | 高（工具查询零AI） | ⭐⭐⭐⭐⭐ |
| 语义缓存 | 中 | 中（需要积累数据） | ⭐⭐⭐⭐ |
| 对话压缩 | 中 | 中（长对话场景） | ⭐⭐⭐ |
| 意图路由（混合） | 高 | 中（覆盖最后10%） | ⭐⭐⭐ |

**策略**：优先实施高ROI的低成本优化，再逐步完善复杂方案。

### 2.3 智能降级理论

任何优化都不能以牺牲**可用性**为代价。系统设计遵循**优雅降级**原则：

```
优化层故障 → 自动降级到下一层 → 最终保底到原始AI调用

语义缓存故障 → 直接查意图路由
意图路由故障 → 直接走AI工具调用
对话压缩故障 → 发送原始消息历史
```

**核心原则**：系统永远可用，只是成本优化程度不同。

## 三、核心层详解

### 3.1 语义缓存层（Semantic Cache）

#### 3.1.1 基本原理

**核心思想**：将用户提问从"字符串匹配"升级为"语义匹配"

传统缓存的局限：
```
精确匹配：
Q: "今天北京天气怎么样？" → 缓存命中
Q: "北京今天天气如何？"   → 缓存未命中（字符串不同）

语义缓存：
Q: "今天北京天气怎么样？" → 向量化 → 存入缓存
Q: "北京今天天气如何？"   → 向量化 → 相似度95% → 命中缓存！
```

**工作流程**：
```
用户提问 → Embedding（文本向量化）→ 向量[0.1, 0.3, 0.5...]
                                              ↓
                                    向量相似度搜索（余弦相似度）
                                              ↓
                                    相似度 > 0.92？
                                              ↓
                              是 → 返回缓存答案（零AI调用）
                              否 → 调AI → 返回答案 → 存入缓存
```

#### 3.1.2 Embedding技术

**概念**：将文本转换为高维空间中的向量，语义相似的文本在向量空间中距离近

**主流开源框架**：

| 框架 | 语言 | 维度 | 特点 | 适用场景 |
|-----|------|------|------|---------|
| **BGE (BAAI)** | 中英 | 768/1024 | 中文优化好，效果SOTA | 生产推荐 |
| **M3E (Moka)** | 中英 | 768 | 轻量，推理快 | 资源有限 |
| **GTE (Alibaba)** | 多语言 | 768 | 长文本支持好 | 长文档 |
| **text2vec** | 中英 | 256/512 | 超轻量，CPU友好 | 边缘设备/快速验证 |

**渐进策略**：
- **开发阶段**：text2vec（轻量，快速验证）
- **生产阶段**：BGE-large-zh（中文效果最佳）

#### 3.1.3 余弦相似度与调优

**公式**：`cos(θ) = (A·B) / (|A|×|B|)`

结果范围：-1（相反）到 1（相同）

**阈值调优**：

| 阈值 | 效果 | 命中率 | 误匹配风险 | 适用场景 |
|-----|------|--------|-----------|---------|
| 0.85 | 宽松 | 高 | 较高 | 客服问答（容错高） |
| **0.92** | **平衡** | **中** | **低** | **一般对话（推荐）** |
| 0.98 | 严格 | 低 | 极低 | 医疗/法律（容错低） |

**调优方向**：

1. **模型选择**：数据量<1万用轻量模型，>10万用大模型
2. **维度压缩**：768维→256维，存储减少67%，速度提升2-3倍
3. **领域微调**：收集领域数据微调，提升专业术语匹配
4. **归一化处理**：确保各维度同等重要，避免大数值维度主导

#### 3.1.4 存储结构

```
Key: semantic_cache:{userId}:{cacheId}
├── query: 原始查询文本
├── response: AI回复答案
├── embedding: 向量[0.1, 0.3, 0.5, ...]
├── userId: 用户ID（用户隔离）
├── createdAt: 创建时间
└── lastAccessed: 最后访问时间（用于TTL）
```

**用户隔离原因**：
- 用户A问"我的日程"→查A的日历
- 用户B问"我的日程"→查B的日历
- 相同问题但答案不同，必须隔离

**TTL策略**：默认7天无访问自动过期

#### 3.1.5 实际效果示例

```
用户首次："今天北京天气怎么样？" → 调AI → 存入缓存
                                    ↓
5分钟后："北京今天天气如何？"   → 向量相似度0.95 → 命中缓存
                                    ↓
又5分钟："查一下北京天气"       → 向量相似度0.91 → 命中缓存
                                    ↓
10分钟后："上海明天天气"        → 向量相似度0.45 → 未命中 → 调AI

成本节省：3次查询中只有1次调AI，节省67%
```

### 3.2 工具缓存层（Tool Cache）

#### 3.2.1 基本原理

**核心思想**：缓存工具调用结果，避免短时间内重复调用外部API

**解决的问题**：
```
用户A：10:00查询"北京天气" → 调天气API → 返回结果
用户A：10:05又查"北京天气" → 又调天气API（浪费！）

优化后：
10:00查询 → 调API → 缓存10分钟 → 返回结果
10:05查询 → 命中缓存 → 直接返回（零API调用）
```

**工作流程**：
```
用户查询工具 → 检查缓存（Redis）
                      ↓
              命中？
                      ↓
        是 → 直接返回缓存结果（零API调用）
        否 → 调工具API → 存入缓存 → 返回结果
```

#### 3.2.2 缓存策略设计

**不同工具的TTL策略**：

| 工具类型 | TTL | 理由 |
|---------|-----|------|
| 天气查询 | 10分钟 | 天气数据变化较快 |
| 日历查询 | 2分钟 | 日程可能随时新增/修改 |
| 待办查询 | 2分钟 | 待办状态可能实时变化 |
| 路线规划 | 5分钟 | 路况可能变化 |

**缓存Key设计**：
```
格式：tool:{toolName}:{paramHash}

示例：
tool:weather:city=北京&date=2024-03-04
tool:calendar:userId=123&start=2024-03-04&end=2024-03-11
```

**数据一致性**：
- 数据变更时主动清除缓存（Calendar/Todo增删改时）
- TTL过期自动清理
- 支持手动清除（管理员接口）

#### 3.2.3 与语义缓存的协同

```
用户："今天北京天气"
      ↓
第1层：语义缓存检查
  有历史相似问题？→ 是 → 直接返回答案（零AI+零API）
                      ↓ 否
第2层：意图路由
  识别为天气查询 → 提取参数（城市=北京，日期=今天）
                      ↓
第3层：工具缓存检查
  该参数有缓存？→ 是 → 返回天气结果（零API）
                      ↓ 否
              调天气API → 存入工具缓存 → 返回答案
                      ↓
              答案存入语义缓存（为下次相似问题准备）
```

**协同优势**：
- 语义缓存处理"同义不同字"的问题
- 工具缓存处理"同一问题短时间内重复"
- 两层配合实现最大化的成本节省

#### 3.2.4 实际效果

| 场景 | 传统方式 | 优化后 | 节省 |
|-----|---------|-------|------|
| 用户10分钟内多次查同一城市天气 | 多次调API | 仅首次调API | 外部API费用省80%+ |
| 页面刷新重复加载 | 重复调API | 命中缓存 | 响应时间从2s→0.1s |
| 批量用户查同一城市天气 | 每人调API | 共享缓存 | API调用次数大幅减少 |

## 四、架构设计方案

### 4.1 总体架构

```
┌─────────────────────────────────────────────────────────────┐
│                         用户请求                             │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  第一层：语义缓存（Semantic Cache）                          │
│  ├─ 向量相似度匹配（余弦相似度 > 0.92）                       │
│  ├─ 用户隔离（只匹配同用户历史）                             │
│  └─ TTL过期（默认7天）                                      │
└─────────────────────────────────────────────────────────────┘
                            ↓ 未命中
┌─────────────────────────────────────────────────────────────┐
│  第二层：三层混合意图路由（Hybrid Intent Router）             │
│  ├─ Layer 1: 规则提取（零AI，85%覆盖）                      │
│  │     关键词匹配 + 正则提取参数                            │
│  ├─ Layer 2: 轻量AI提取（1轮AI，10%覆盖）                    │
│  │     极简提示词仅提取参数（maxTokens=100）                 │
│  └─ Layer 3: 完整AI工具（保底，5%覆盖）                      │
│        标准MCP工具调用流程                                   │
└─────────────────────────────────────────────────────────────┘
                            ↓ 复杂对话
┌─────────────────────────────────────────────────────────────┐
│  第三层：对话压缩（Conversation Compression）                │
│  ├─ 触发条件：消息数 > 10                                    │
│  ├─ 保留最近4条原始消息                                      │
│  └─ 早期消息压缩为摘要（AI生成100字摘要）                     │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  第四层：AI调用（AI Call）                                    │
│  └─ 真正需要AI创造力的对话                                   │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 三层混合意图路由详解

这是本方案的核心创新点，解决**"提取失败是否必须用2轮AI"**的问题。

#### 4.2.1 传统方案的问题

```
用户：这里天气怎么样？
      ↓
规则提取：失败（"这里"不是城市名）
      ↓
只能走：2轮完整AI工具调用 ❌
```

#### 4.2.2 三层混合方案架构

```
用户提问："这里天气怎么样？"
      ↓
┌─────────────────────────────────────────┐
│ 第1层：规则提取（Rule-based）            │
│ 覆盖：85%的常见表达                      │
│ 成本：零AI                               │
│ 原理：关键词匹配 + 正则提取参数          │
└─────────────────────────────────────────┘
              ↓ 提取失败
┌─────────────────────────────────────────┐
│ 第2层：轻量AI提取（Lightweight AI）      │
│ 覆盖：10%的模糊表达                      │
│ 成本：1轮极简AI（200 token）             │
│ 原理：极简提示词仅提取参数               │
└─────────────────────────────────────────┘
              ↓ 提取失败
┌─────────────────────────────────────────┐
│ 第3层：完整AI工具（Full AI）             │
│ 覆盖：5%的复杂表达                       │
│ 成本：2轮完整AI（1500 token）            │
│ 原理：标准MCP工具调用流程                │
└─────────────────────────────────────────┘
```

#### 4.2.3 各层详细说明

**第1层：规则提取**

```java
// 意图分类
用户问题 → 关键词匹配 → 意图类型

"北京天气"     → 命中"天气"关键词 → WEATHER_QUERY
"我今天有什么会" → 命中"会/日程"    → CALENDAR_QUERY
"帮我记个待办"   → 命中"待办"       → TODO_CREATE

// 参数提取规则
天气查询参数：
- 城市（必需）：匹配预定义城市列表
- 日期（可选）：今天/明天/后天 → 转换具体日期

提取示例：
"北京天气"           → 城市=北京，日期=今天
"查深圳明天天气"     → 城市=深圳，日期=明天
"天气怎么样"         → 城市=默认(北京)
```

**第2层：轻量AI提取**

为什么需要这一层？规则无法覆盖模糊表达：
```
规则能处理："北京天气" ✅
规则不能处理："这里天气" ❌（"这里"是哪里？）
              "所在城市天气" ❌（需要上下文）
```

**轻量AI vs 完整AI对比**：

| 维度 | 轻量AI提取 | 完整AI工具调用 |
|-----|-----------|--------------|
| **任务** | 只提取参数 | 理解+调用+生成回复 |
| **提示词** | 极简（50字） | 复杂（500+字） |
| **输出** | JSON格式参数 | 自然语言回复 |
| **调用轮数** | 1轮 | 2轮 |
| **Token消耗** | ~200 | ~1500 |
| **成本** | ¥0.02 | ¥0.30 |

**轻量AI工作流程**：
```
用户：这里天气怎么样？
      ↓
构建极简提示词：
  "从以下查询中提取城市名，只返回JSON格式：
   查询: '这里天气怎么样'
   用户位置: 北京
   返回格式: {\"city\": \"城市名\"}"
      ↓
调用AI（maxTokens=100，temperature=0.1）
      ↓
AI返回：{"city": "北京"}
      ↓
提取参数 → 调天气API(北京) → 返回结果
```

**关键优化**：
- `maxTokens=100`：强制AI只输出参数，不废话
- `temperature=0.1`：低随机性，结果稳定
- 结合用户上下文（如用户常住北京）

**第3层：完整AI工具**

当轻量AI也提取失败时，回退到原始方案：
```
用户：离我最近的海边城市天气
      ↓
规则提取：失败（太复杂）
轻量AI提取：失败（需要地理计算）
      ↓
完整AI工具调用：
  第1轮：AI理解需要调用天气工具+解析复杂语义
  执行：地理计算+天气查询
  第2轮：AI生成自然语言回复
```

**为什么保留这一层？**
- 确保任何查询都能被处理
- 复杂场景AI的理解能力无可替代
- 只有<5%的查询会走到这一层

#### 4.2.4 三层决策流程示例

```
用户提问："这里天气怎么样？"
      ↓
Step 1: 意图分类
  关键词匹配"天气" → WEATHER_QUERY
      ↓
Step 2: 第1层-规则提取
  尝试提取城市名
  "这里"不在城市列表 → 失败 ❌
      ↓
Step 3: 第2层-轻量AI提取
  调用轻量AI，结合用户上下文
  返回：{"city": "北京"} → 成功 ✅
      ↓
Step 4: 执行工具调用
  调天气API(北京) → 获取结果
      ↓
Step 5: 格式化返回
  模板："北京今天{天气}，{温度}..."
      ↓
返回给用户

成本统计：
- 意图分类：零成本（关键词匹配）
- 参数提取：1轮轻量AI（¥0.02）
- 工具执行：调天气API（零AI成本）
- 总计：¥0.02（vs 完整AI的¥0.30，省93%）
```

#### 4.2.5 与传统方案对比

假设处理1000个天气查询：

| 方案 | 规则覆盖率 | 处理方式 | 总成本 |
|-----|-----------|---------|--------|
| **传统AI工具** | 0% | 全部2轮AI | ¥300 |
| **纯规则** | 85% | 850个零AI，150个失败 | ¥45 |
| **三层混合（推荐）** | **95%** | 850个零AI+100个轻量AI+50个完整AI | **¥24** |

## 五、技术选型思路

### 5.1 向量数据库选型

| 方案 | 适用规模 | 查询复杂度 | 运维成本 | 推荐场景 |
|-----|---------|-----------|---------|---------|
| **Redis Hash（简化版）** | <1万条 | O(n)遍历 | 极低 | 开发/测试 |
| **Chroma** | <100万条 | O(log n) HNSW | 低 | 中小生产 |
| **Milvus** | >100万条 | O(log n) 分布式 | 中 | 大规模生产 |
| **pgvector** | 不限 | O(log n) | 低 | 已有PostgreSQL |

### 5.2 近似搜索算法选型（大数据量优化）

当向量数据量>100万，暴力计算O(n)太慢：

| 算法 | 时间复杂度 | 召回率 | 适用规模 | 特点 |
|-----|-----------|--------|---------|------|
| 暴力搜索 | O(n) | 100% | <10万 | 精准但慢 |
| **HNSW** | O(log n) | 95%+ | 推荐 | 构建索引慢，查询快 |
| IVF | O(log n) | 90%+ | 大规模 | 内存友好 |
| LSH | O(1) | 80%+ | 超大规模 | 最快但精度低 |

**推荐**：HNSW（Hierarchical Navigable Small World）
- 查询时毫秒级返回
- 召回率>95%（几乎不损失准确性）
- Chroma/Milvus原生支持

**选型原则**：
1. **开发阶段**：Redis简化版，零额外依赖
2. **数据增长后**：平滑迁移到Chroma/Milvus
3. **接口统一**：VectorStore抽象层屏蔽底层差异

### 5.2 嵌入（Embedding）方案

| 方案 | 维度 | 准确性 | 成本 | 适用场景 |
|-----|------|-------|------|---------|
| **关键词One-Hot** | 100维 | 中 | 零 | 快速验证/MVP |
| **API嵌入（OpenAI/Kimi）** | 1536维 | 高 | 中 | 生产环境 |
| **本地模型（BGE/M3E）** | 768维 | 高 | 低（GPU） | 高频查询 |

**渐进策略**：
- 初期：关键词方案快速验证
- 中期：API嵌入提升准确性
- 长期：本地模型降低成本

### 5.3 缓存存储选型

| 数据类型 | 存储方案 | TTL策略 |
|---------|---------|---------|
| 语义缓存 | Redis/向量库 | 7天无访问过期 |
| 工具结果 | Redis | 天气10分钟/日历2分钟 |
| 成本统计 | MySQL | 永久保存 |
| 会话状态 | Redis | 会话结束时过期 |

## 六、优化方向与预期效果

### 6.1 各层优化效果

#### 语义缓存层
- **命中场景**："今天北京天气" ↔ "北京今天天气如何"
- **相似度算法**：余弦相似度 > 0.92
- **预期命中率**：20-30%（随数据积累增长）
- **成本节省**：100%（零AI调用）

**调优方向**：
- **模型选择**：数据量<1万用轻量模型（text2vec），>10万用大模型（BGE）
- **维度压缩**：768维→256维，存储减少67%，速度提升2-3倍
- **领域微调**：收集领域数据微调，提升专业术语匹配
- **阈值调整**：0.85（宽松）→ 0.92（推荐）→ 0.98（严格）

#### 工具缓存层
- **TTL策略**：天气10分钟 / 日历2分钟 / 待办2分钟
- **缓存Key**：tool:{toolName}:{paramHash}
- **一致性保障**：数据变更时主动清除 + TTL过期 + 手动清除
- **成本节省**：外部API调用减少80%+

#### 意图路由层（三层混合）
| 场景占比 | 处理方式 | AI调用 | 成本对比 |
|---------|---------|--------|---------|
| 85% | 规则提取 | 0轮 | 省100% |
| 10% | 轻量AI提取 | 1轮（200token） | 省80% |
| 5% | 完整AI工具 | 2轮（1500token） | 基准 |

**加权平均成本**：仅为传统方案的 **8-10%**

#### 对话压缩层
- **触发条件**：消息数 > 10
- **压缩策略**：早期消息 → 100字摘要
- **token节省**：长对话场景减少50-60%

### 6.2 整体效果预估

假设日活1000用户，每人10次交互：

| 指标 | 优化前 | 优化后 | 提升 |
|-----|-------|-------|------|
| 日均AI调用 | 10,000次 | 1,000次 | **-90%** |
| 平均响应时间 | 3秒 | 0.8秒 | **-73%** |
| 月AI成本 | ¥3,000 | ¥300 | **-90%** |
| 用户体验 | 可接受 | 流畅 | **显著提升** |

### 6.3 长期优化方向

#### 阶段1：基础优化（当前）
- 工具缓存 + 规则意图路由
- 目标：80%工具查询零AI

#### 阶段2：智能增强（3个月后）
- 语义缓存积累数据 + 轻量AI提取

## 七、实施建议

### 7.1 优先级排序

**P0 - 立即实施（高ROI，低风险）**
1. 工具结果缓存（天气/日历/待办）- 效果最明显，零风险

3. 规则意图路由（关键词匹配）- 零AI成本
4. 系统提示词精简 - 简单有效

**P1 - 短期实施（中ROI，中风险）**
1. 语义缓存升级（BGE Embedding）- 提升准确性
2. 轻量AI提取层（三层混合）- 覆盖最后10%
3. 对话历史压缩 - 长对话场景优化
4. 向量数据库升级（Chroma）- 支持更大规模

**P2 - 长期完善（监控运维）**

## 八、总结

### 8.1 核心创新点

1. **三层混合意图路由**：不是简单的"规则失败就调AI"，而是引入**轻量AI提取层**，用10%的成本解决最后10%的复杂场景。

2. **渐进式优化架构**：每层都可独立开关，支持从简单规则到智能混合的平滑演进。

3. **零侵入集成**：通过AOP和装饰器模式，对现有代码零修改。

### 8.2 关键设计决策

| 决策 | 选择 | 理由 |
|-----|------|------|
| 意图路由模式 | 三层混合 | 平衡成本与覆盖率的最佳点 |
| Embedding方案 | 关键词One-Hot起步 | 快速MVP，后期升级BGE |
| 向量存储 | Redis简化版起步 | 零额外依赖，快速验证 |
| 降级策略 | 每层独立降级 | 确保系统永远可用 |

### 8.3 预期成果

- **成本**：AI调用减少90%，月成本从¥3000降至¥300
- **性能**：平均响应时间从3秒降至0.8秒
- **体验**：用户感知到的响应更快更稳定
- **运维**：完善的监控体系，实时掌握成本与效果
